The problem

Counterfeits don't announce themselves.

Fake medicine, drinks, cosmetics, agro-inputs and electronics sit on the same shelves as the genuine article, in the same packaging, at the same price. By the time a customer notices something's wrong, the damage — to their health, or to your brand — is already done.

A hologram sticker can be copied. A serial number printed on a box can be reused. What can't be faked as easily is a system that checks a code's entire history the instant someone asks about it.

Same shelf

Counterfeit and genuine units are sold side by side, often without either party knowing.

Copied marks

Static stickers, holograms and printed serials can be photographed and reproduced.

Silent losses

Brands often learn about counterfeiting from a complaint, not from their own data.

No easy check

Consumers have no simple, trusted way to confirm authenticity before they buy or use a product.

Questions

Before you ask, we probably already have.

Does my customer need to install an app?

No. Scanning the code opens a normal web page. The whole flow — phone number, OTP, result — happens in the browser they already have.

What happens if someone scans a genuine code twice?

A second scan of the same code isn't automatically treated as fraud. It's shown as "Previously Verified," and the risk engine looks at how, where and how often it's being repeated before deciding whether it's normal or worth flagging.

What if a customer doesn't have a smartphone with a camera?

The code can be typed into the verification page manually, and the OTP step still works over a basic SMS-capable phone.

How do you catch counterfeits without inspecting the physical product?

By treating every scan as a data point, not just a lookup. Location, device, timing and repetition patterns across the whole platform reveal counterfeit activity that a single valid/invalid check would miss.

Can AntiFakeNG run several brands under one account?

Yes. That's the core of the platform — an administrator can onboard multiple producers, each with its own products, users and data boundary, while still seeing fraud patterns across the whole platform.

Vision & objectives

A multi-tenant SaaS enabling manufacturers to generate secure authenticity codes, verify products through a browser, detect counterfeit activity, and gain intelligence on product movement without requiring a mobile application.

Primary objectives

  • — Give every physical unit a verifiable digital identity.
  • — Let any consumer confirm authenticity without installing software.
  • — Convert every verification into supply-chain intelligence for the brand.
  • — Detect and localize counterfeit activity as it happens, not after a complaint.
  • — Operate many brands securely on one shared, multi-tenant platform.

Core value proposition

  • — Unique QR per individual product, never reused.
  • — Browser-based verification, no app required.
  • — Phone-number + OTP identity binding on every check.
  • — A risk engine, not simple valid/invalid logic.
  • — Multi-brand platform operated by one SaaS provider.

Fraud detection engine

Each verification is scored against a weighted set of signals rather than judged on a single rule.

SignalWeightDescription
Tampered tokenCriticalThe code's signature fails verification.
Unknown QRCriticalThe token does not exist in the system at all.
Impossible travelHighRelated tokens verified from geographically distant locations within an implausible window.
Different phone number reusedHighThe same phone verifies an unusual number of distinct products in a short window.
Device mismatchMediumOne code verified from many distinct device fingerprints.
Country mismatchMediumVerification originates outside the product's expected distribution region.
Multiple scans in short periodMediumA single code verified repeatedly beyond typical consumer behavior.
High-volume repeated scansMediumBatch-level scan volume spikes beyond historical baseline.
Known VPN or proxyLow–MediumThe verification's network origin is obscured.

Verdicts

Genuine

Valid signature, first verification, no signals raised.

Previously Verified

Valid code, verified before, within expected repeat behavior.

Suspicious

Valid code, one or more medium-weight signals raised.

High Risk

Valid code, multiple signals or one high-weight signal raised.

Invalid

Token unknown or signature tampered.

Recalled / Blocked

Code belongs to a recalled batch, or has been manually blocked.

Security & compliance

Baseline controls

  • — HTTPS enforced everywhere, no plain-HTTP fallback.
  • — Encryption at rest and in transit for sensitive fields.
  • — Consumer phone numbers stored hashed, never in plain text.
  • — Signed, time-boxed verification tokens.
  • — Rate limiting on OTP, verification and API endpoints.
  • — Row-level isolation between tenants at the database layer.

Data protection posture

Consumer data handling is designed with the Nigeria Data Protection Act (NDPA) in mind: minimal collection, hashed storage of identifiers, defined retention periods, and an audit trail of who accessed what. A formal compliance review and any relevant registration should be completed with qualified counsel before handling personal data at scale.

Operational practice

  • — Independent penetration testing before major releases.
  • — Documented data retention and deletion policy.
  • — Immutable audit logs, exportable for review.
